# Как работи storage

### Основното гарантиране

Всичко, което се съхранява в базата данни, е криптирано. Сървърът държи по два блоба за всякa кола:

- `encrypted_embedding` - CKKS-криптиран 256-мерен вектор с плаваща запетая  
- `encrypted_metadata` - JSON метаданни (регистрационен номер, цвят, тип купе, път до изображение) кодиран като байтове и криптиран като CKKS вектор

Секретният ключ никога не напуска клиента. Сървърът не може да декриптира нито един от блобовете.

### Вмъкване (Insert)

Когато се вмъква превозно средство чрез `insert_vehicle`:

1. 256-мерният вектор се L2-нормализира (така косинусната прилика се свежда до скаларно произведение)
2. както векторът, така и JSON метаданните се CKKS-криптират на страна на клиента
3. в базата данни се записват само криптограмните блобове

### Търсене (PIR)

`pir_search_true` извършва пълно хомоморфно търсене:

1. Клиентът нормализира и криптира заявъчния вектор (`PIRClient.prepare_query`)
2. Сървърът извлича всички криптирани записи и изчислява за всеки `enc_query · enc_db_vec`, използвайки хомоморфно скаларно произведение (TenSEAL) — изцяло върху криптограми
3. Сървърът връща всички N криптирани сходствени оценки, без да ги декриптира
4. Клиентът декриптира всички N оценки локално, сортира и избира топ‑K
5. Клиентът декриптира само топ‑K метадата локално

Сървърът винаги сканира всички записи при всяка заявка. Това е умишлено — така сървърът не може да извлече кои записи са съвпаднали от моделите на достъп.

### Декриптиране на метаданни

Метаданните се съхраняват като CKKS-криптирани UTF‑8 байтове, мащабирани в `[0, 1]` преди криптиране и възстановявани при декрипция:

```
encrypt: byte_values / 255.0  ->  CKKS vector
decrypt: CKKS vector * 255.0  ->  clamp  ->  round  -> bytes  ->  JSON
```

CKKS въвежда малък плаващ шум (~10⁻⁷) при хомоморфните операции. Стъпките `clip` + `round` компенсират този шум, когато оригиналните байтове са в границите `[0,255]`.

## Архитектура

```
database/
├── connection.py     # SQLAlchemy session и engine конфигурация
├── models.py         # Модел Vehicle (id, uuid, encrypted_embedding, encrypted_metadata)
├── operations.py     # insert_vehicle, pir_search_true, get_all_vehicles, stats
└── encryption.py     # CKKSEncryptor, homomorphic_dot_product, PIRClient,
                      # encrypt/decrypt embedding и metadata
```

CKKS контекстът (полиномен ред 8192, размери на коефициентите `[60, 40, 40, 60]`, глобален скейл 2⁴⁰) се създава веднъж и се запазва в `ckks_context.bin`. Последващи стартирания го зареждат от диск, за да запазят съвместимостта на криптограмите.

## Производителност (16 записа, локално)

| Операция                               | Време    |
|----------------------------------------|----------|
| Подготовка на заявка (криптиране)      | ~11 ms   |
| Сървърно сканиране + хом. произведения | ~560 ms  |
| Клиентско декриптиране + сортиране     | ~220 ms  |
| **Общо търсене**                       | **~800 ms** |

Времето за търсене нараства линейно с броя записи - сървърът винаги прави пълен скан.

### Нужда за място (Storage overhead)

| Показател                 | Стойност       |
|---------------------------|----------------|
| Размер на плейнтекст вектор | 1,024 байта (256 × float32) |
| Среден размер на криптиран запис | ~668 KB       |
| Нарастване на размера     | ~653×          |

Това е цената на раздуването при CKKS. За полиномен ред 8192 и 200‑битов коефициентен модул, криптограмите стават големи — стандартен компромис за поддръжка на хомоморфни операции върху пакетирани вектори.

## Свойства на сигурността

| Свойство                                 | Статус |
|------------------------------------------|--------|
| Сървърът вижда плейнтекст вектори        | Не     |
| Сървърът вижда плейнтекст метаданни      | Не     |
| Сървърът научава заявъчния вектор        | Не     |
| Сървърът научава кой запис е съвпаднал   | Не - сканират се всички N записи всяка заявка |
| Секретният ключ напуска клиента          | Не - съхранява се само в `ckks_context.bin` |
| Изтичане чрез модели на достъп (access pattern) | Минимално - сървърът вижда N униформни достъпа за всяка заявка |

Какво не е това: не е теоретично PIR с информация без вероятности (което би изисквало множество некорелиращи се сървъри или ORAM). Това е изчислително PIR — сигурността се базира на трудността на RLWE, която стои зад CKKS. Атакуващ, който притежава само криптограмните блобове и няма секретния ключ, не може да възстанови плейнтекстите.

## Зависимости

- [TenSEAL](https://github.com/OpenMined/TenSEAL) — CKKS хомоморфно криптиране  
- [SQLAlchemy](https://www.sqlalchemy.org/) - ORM и управление на PostgreSQL сесии  
- PostgreSQL с разширение `pgcrypto` (`gen_random_uuid()`)  
- NumPy

## Стартиране на тестовете

```bash
python -m pytest test.py -v
```

Тестовият пакет (`test.py`) покрива:

1. Вмъкване на 8 криптирани превозни средства  
2. Статистика и анализ на съхранението  
3. Хомоморфно PIR търсене с подробно отчитане на времето  
4. Валидация на декриптацията на метаданните за всички top‑K резултати  
5. Няколко изпълнения за оценка на производителността при търсене  
6. Анализ на използваното място за съхранение

Всички полета - embedding, metadata, query - остават криптирани през целия процес. Тестът валидира коректността чрез клиентско декриптиране след извличане.
